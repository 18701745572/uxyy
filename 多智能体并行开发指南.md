# 优效营（uxyy.cn）多智能体并行开发指南

> 本文档指导如何基于当前 PRD 进行多智能体（Multi-Agent）并行开发。**工程默认值**：pnpm Monorepo + Turborepo + NestJS + Passport JWT + Next.js 14 + Drizzle（Neon 兼容 Postgres）+ Redis + BullMQ + `@uxyy/shared`（Zod）；与《产品需求文档》不一致时以 PRD 修订为准。

---

## 一、开发前准备

### 1.0 工程默认值（并行开发最全栈）

本仓库以 **pnpm Monorepo + Turborepo** 为底座，与各智能体并行开发对齐；以下为定版默认组合（认证为 **Passport JWT**）。

| 层级 | 选型 | 说明 |
|------|------|------|
| 仓库 / 编排 | pnpm workspaces + Turborepo | 根目录统一脚本；`pnpm -w` / `pnpm --filter @uxyy/api` |
| 前端 | Next.js 14 App Router + TS + Tailwind | 目录 `uxyy-web`（建议包名 `@uxyy/web`） |
| 后端 | NestJS + TS | 目录 `uxyy-api`（建议包名 `@uxyy/api`）；模块边界与 `@Module()` 对齐 |
| 契约 | `@uxyy/shared` + Zod + OpenAPI | 共享校验与类型；Swagger 由 `@nestjs/swagger` 维护并与契约一致 |
| 数据库 | Neon（或等价 PostgreSQL）+ Drizzle ORM | 迁移仅用 `drizzle-kit`，禁止手写漂移 |
| 认证 | **Passport JWT**（`@nestjs/passport` + `passport-jwt`）+ JwtStrategy | 受保护路由使用 `AuthGuard('jwt')`（或等价封装）；Access / Refresh 策略见 PRD |
| 缓存 / 队列 | Redis + BullMQ | 热点缓存、限流、令牌黑名单辅助；异步任务统一走队列 Worker |

**Nest 模块与智能体默认目录映射（不得在模块间逆向依赖领域层）：**

| 智能体 | 建议 Nest 路径 |
|--------|----------------|
| Agent-Auth | `uxyy-api/src/modules/auth/` |
| Agent-CRM | `uxyy-api/src/modules/crm/` |
| Agent-Inventory | `uxyy-api/src/modules/inventory/` |
| Agent-Finance | `uxyy-api/src/modules/finance/` |
| Agent-AI | `uxyy-api/src/modules/ai/` |

## 二、Phase 0：基础定版（第1周）

> **目标**：所有智能体共同完成项目基础架构、数据库Schema定版、接口契约定义。

### 2.1 数据库Schema定版流程

**步骤 1：架构师定义初始Schema**
```powershell
# 在 develop 分支创建 Schema 定义
git checkout develop
git pull origin develop

# 创建 Drizzle Schema 文件
New-Item -ItemType Directory -Force -Path uxyy-api/src/db/schema | Out-Null
New-Item -ItemType File -Force -Path uxyy-api/src/db/schema/index.ts | Out-Null
```

**步骤 2：Schema 评审会议**
- 所有智能体参与评审
- 确认表结构、字段类型、索引设计
- 确认表所有权（见 PRD 11.5.2 节）

**步骤 3：定版并锁定**
```powershell
# 生成初始 Migration（在 uxyy-api 中已配置 drizzle.config 的前提下）
cd uxyy-api
pnpm exec drizzle-kit generate --name init_schema
cd ..

# 提交并合并到 develop
git add .
git commit -m "schema: 定版 MVP 数据库Schema v1.0.0"
git push origin develop
```

**定版后规则：**
- 共享表（users, enterprises, roles）**禁止修改**
- 模块私有表由对应智能体维护
- 所有变更必须通过 Migration 文件
- 使用 **Neon** 时，建议为 `develop`/PR 预览配置**独立分支库**，勿直接对 production branch 试错

### 2.2 接口契约定义流程

> **说明**：定版契约的「机器可读来源」可为 `docs/api/*.yaml`，同时 **NestJS** 侧须用 **`@nestjs/swagger` + DTO（`@ApiProperty` 等）** 声明入参/出参，保证运行时可访问的 Swagger 与契约、**`@uxyy/shared` 中 Zod** 对齐（任一变更须同步其余二者）。

**步骤 1：创建 OpenAPI 规范文件**
```powershell
New-Item -ItemType Directory -Force -Path docs/api | Out-Null
New-Item -ItemType File -Force -Path docs/api/swagger-v1.0.0.yaml | Out-Null
```

**步骤 2：定义核心接口契约**
```yaml
# docs/api/swagger-v1.0.0.yaml
openapi: 3.0.0
info:
  title: 优效营 API
  version: 1.0.0
  description: MVP阶段接口契约（已定版）

paths:
  /api/v1/auth/register:
    post:
      summary: 用户注册
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [phone, password, enterpriseName]
              properties:
                phone: { type: string, pattern: '^1[3-9]\d{9}$' }
                password: { type: string, minLength: 6 }
                enterpriseName: { type: string, maxLength: 100 }
      responses:
        200:
          description: 注册成功
          content:
            application/json:
              schema:
                type: object
                properties:
                  code: { type: integer, example: 200 }
                  data:
                    type: object
                    properties:
                      userId: { type: integer }
                      accessToken: { type: string }
```

**步骤 3：契约锁定**
```powershell
git add docs/api/swagger-v1.0.0.yaml
git commit -m "docs(api): 定版接口契约 v1.0.0"
git push origin develop
```

### 2.3 环境配置共享（`@uxyy/shared`）

**workspace 共享包推荐布局：**
```powershell
New-Item -ItemType Directory -Force -Path uxyy-shared/src/schemas | Out-Null
New-Item -ItemType Directory -Force -Path uxyy-shared/src/types | Out-Null
New-Item -ItemType Directory -Force -Path uxyy-shared/src/constants | Out-Null
```

**约定**：DTO 与 Swagger、`zod` Schema 字段名保持一致；Nest 使用 `class-validator`，请求体验证可与 Zod 推导类型交叉检查。

**示例（TypeScript 文件，非 PowerShell）**：`uxyy-shared/src/schemas/pagination.ts`
```typescript
import { z } from 'zod';
export const paginationSchema = z.object({
  page: z.number().min(1),
  pageSize: z.number().max(100),
});
export type PaginationParams = z.infer<typeof paginationSchema>;
```

### 2.4 环境与密钥（最全栈默认值）

**所有智能体共用同一套变量名（示例，按环境切分 `.env` / 托管平台 Secret）：**

| 变量 | 用途 |
|------|------|
| `DATABASE_URL` | **Neon** 或本地 Postgres 连接串（Drizzle）；生产须与连接池/Neon 文档一致 |
| `REDIS_URL` | 缓存、Passport 登出黑名单辅助、**BullMQ** |
| `JWT_SECRET` / `JWT_REFRESH_SECRET` | `@nestjs/jwt` 签发；与 Passport 校验用的 `secretOrKey` 配置一致 |
| `PORT` | Nest 监听端口（默认常设 `3001`，与 Next `3000` 区分） |

**约定**：密钥不入库；根或各包提供 `.env.example` 列举键名；本地可用 §7.3 `docker-compose` 中的 Postgres + Redis 替代云上实例。

### 4.3 版本升级规则

| 变更类型 | 版本号变化 | 示例 | 是否需要协商 |
|---------|-----------|------|-------------|
| 新增可选字段 | 修订号+1 | v1.0.0 → v1.0.1 | 否 |
| 新增接口 | 次版本+1 | v1.0.0 → v1.1.0 | 否 |
| 新增必填字段 | 次版本+1 | v1.0.0 → v1.1.0 | 是 |
| 删除字段/接口 | 主版本+1 | v1.0.0 → v2.0.0 | 必须 |
| 修改字段类型 | 主版本+1 | v1.0.0 → v2.0.0 | 必须 |

**步骤 2：编写 Migration 文件**


**步骤 3：验证 Migration**

### 6.4 并行开发门禁清单（契约 / Migration / 模块边界）

> **目标**：把「按契约演进、按 Migration 改库、按模块边界分工」变成**可检查、可拦阻、可追溯**的习惯；与 §2、§4、§5 及 Phase 0 相互配合。

#### 契约（HTTP / DTO）

| 类型 | 要求 |
|------|------|
| **PR 自检** | 凡变更请求体/响应体/错误码：PR 须勾选「接口契约变更」，写明兼容性与受影响智能体 |
| **三处同步** | 涉及对外行为的改动：`docs/api` YAML（若保留定版）、`@nestjs/swagger` DTO、**若存在对应 Zod**：`@uxyy/shared`，须在同一 PR 内同步，或可拆 PR 但须在描述中写明**合并顺序** |
| **Code Review** | 驳回「只改控制器而不同步 DTO / Zod / YAML / Swagger 任一」的单点提交 |
| **递进（可选）** | OpenAPI 破坏性对比、契约测试，或基于 Swagger **生成前端类型**，减少手写漂移 |

#### Migration（数据库）

| 类型 | 要求 |
|------|------|
| **独立 PR** | Schema 演进走 `migration/*`（或团队统一前缀），禁止与大批量业务改写混在同一 PR（见 §5） |
| **合并顺序** | 先合并 Migration PR；业务代码在最新 **develop** 上再继续 |
| **CI（建议）** | Migration PR：在**隔离 Postgres**（或 Neon **非 production** 分支）执行 `pnpm exec drizzle-kit migrate`（或项目脚本），失败则不可合并 |
| **高风险变更** | 共享表结构与关键索引：**架构师审批**或 **CODEOWNERS** 强制 Review |

#### 模块边界（Nest 域划分）

| 类型 | 要求 |
|------|------|
| **Nest 模块化** | 域内实现在 `src/modules/<域>/`；跨域只允许依赖对方 **刻意 export 的模块/服务**，禁止深层引用他域「内部」文件 |
| **`@uxyy/shared` 守则** | 仅放：**Zod、常量、无业务副作用的类型与小幅工具**；不放领域业务流程，避免共享包臃肿 |
| **递进（可选）** | `eslint-plugin-boundaries`、`dependency-cruiser` 等禁止违规 import |

#### 与人的配合

| 类型 | 要求 |
|------|------|
| **合并门禁** | 合并前完成 §6.2 清单；涉及契约/Migration 的 PR 由约定 Owner 终审 |
| **事故复盘** | 契约或 Migration 导致联调阻塞：记在 Issue/Discussion **一条改进**（规则或 CI），避免重复踩坑 |

本节与 **§10.4（BullMQ 队列命名约定）** 同属并行纪律：**契约、迁移、队列、模块边界** 任一类失守都会拖慢其他 Agent。

```yaml
on:
  pull_request:
    branches: [develop, main]

  detect-agent:
    runs-on: ubuntu-latest
    outputs:
      agent: ${{ steps.detect.outputs.agent }}
    steps:
      - uses: actions/checkout@v4
      - id: detect
        run: |
          BRANCH="${{ github.head_ref }}"
          if [[ $BRANCH == feature/auth* ]]; then 
            echo "agent=auth" >> $GITHUB_OUTPUT
          elif [[ $BRANCH == feature/crm* ]]; then 
            echo "agent=crm" >> $GITHUB_OUTPUT
          elif [[ $BRANCH == feature/inventory* ]]; then 
            echo "agent=inventory" >> $GITHUB_OUTPUT
          elif [[ $BRANCH == feature/finance* ]]; then 
            echo "agent=finance" >> $GITHUB_OUTPUT
          elif [[ $BRANCH == feature/ai* ]]; then 
            echo "agent=ai" >> $GITHUB_OUTPUT
          elif [[ $BRANCH == feature/frontend* ]]; then 
            echo "agent=frontend" >> $GITHUB_OUTPUT
          elif [[ $BRANCH == migration/* ]]; then
            echo "agent=migration" >> $GITHUB_OUTPUT
          else
            echo "agent=mono" >> $GITHUB_OUTPUT
          fi

  backend-quality:
    needs: detect-agent
    if: needs.detect-agent.outputs.agent != 'frontend'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4

          version: 9
      - uses: actions/setup-node@v4

          node-version: 20

      - run: pnpm install --frozen-lockfile
      - run: pnpm run lint
      - run: pnpm run typecheck
      - run: pnpm run test:coverage

  frontend-quality:
    needs: detect-agent
    if: needs.detect-agent.outputs.agent == 'frontend'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 9
      - uses: actions/setup-node@v4

          node-version: 20
          cache: 'pnpm'

      - run: pnpm run lint

      - run: pnpm run test:coverage
```

### 9.3 数据库 Migration 冲突

**解决方案：**
1. 使用时间戳命名替代序号（Drizzle 默认）
2. 架构师协调合并顺序
3. 冲突时重新生成 Migration

```powershell
cd uxyy-api
pnpm exec drizzle-kit check
pnpm exec drizzle-kit generate --name migration_v2
```

## 十、最佳实践

### 10.1 沟通规范

| 场景 | 沟通方式 | 示例 |
|------|---------|------|
| 接口变更 | PR 评论 + @mention | `@agent-frontend 接口新增字段` |
| 紧急问题 | GitHub Issue | 创建 Bug Issue 并标记 P0 |
| 日常同步 | 每日站会（文字） | 在 Discussion 更新进度 |
| 架构决策 | GitHub Discussion | 创建讨论帖征集意见 |

### 10.2 代码审查清单

**审查他人 PR 时检查：**
- [ ] 代码是否符合编码规范
- [ ] 是否有足够的单元测试
- [ ] 是否影响其他模块
- [ ] 接口契约（YAML / Swagger / Zod）是否一致
- [ ] 数据库变更是否合理
- [ ] 是否有安全漏洞

### 10.3 性能优化建议

- **数据库查询**：添加必要的索引，避免 N+1 查询
- **API 响应**：使用 Redis 缓存热点数据；耗时操作入 **BullMQ** 队列由 Worker 异步执行
- **前端加载**：使用 Next.js 自动代码分割
- **图片资源**：使用 WebP 格式，CDN 加速

### 10.4 BullMQ / Redis 协作（多 Agent 约定）

- **默认**：短时任务可由 `uxyy-api` 进程内挂载 BullMQ Processor；耗时或隔离需求高时，拆分为同仓 **`uxyy-worker` 包**（或等价目录）单独部署。
- **PR 必填说明**：队列名前缀、幂等键、并发与重试策略，避免多人创建同名队列或重复消费路径。
