## 二、Phase 0：基础定版（第1周）

> **目标**：所有智能体共同完成项目基础架构、数据库Schema定版、接口契约定义。

### 2.1 数据库Schema定版流程

**步骤 1：架构师定义初始Schema**
```powershell
# 在 develop 分支创建 Schema 定义
git checkout develop
git pull origin develop

# 创建 Drizzle Schema 文件
New-Item -ItemType Directory -Force -Path uxyy-api/src/db/schema | Out-Null
New-Item -ItemType File -Force -Path uxyy-api/src/db/schema/index.ts | Out-Null
```

**步骤 2：Schema 评审会议**
- 所有智能体参与评审
- 确认表结构、字段类型、索引设计
- 确认表所有权（见 PRD 11.5.2 节）

**步骤 3：定版并锁定**
```powershell
# 生成初始 Migration
npx drizzle-kit generate:pg --name init_schema

# 提交并合并到 develop
git add .
git commit -m "schema: 定版 MVP 数据库Schema v1.0.0"
git push origin develop
```

**定版后规则：**
- 共享表（users, enterprises, roles）**禁止修改**
- 模块私有表由对应智能体维护
- 所有变更必须通过 Migration 文件

### 2.2 接口契约定义流程

**步骤 1：创建 OpenAPI 规范文件**
```powershell
New-Item -ItemType Directory -Force -Path docs/api | Out-Null
New-Item -ItemType File -Force -Path docs/api/swagger-v1.0.0.yaml | Out-Null
```

**步骤 2：定义核心接口契约**
```yaml
# docs/api/swagger-v1.0.0.yaml
openapi: 3.0.0
info:
  title: 优效营 API
  version: 1.0.0
  description: MVP阶段接口契约（已定版）

paths:
  /api/v1/auth/register:
    post:
      summary: 用户注册
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [phone, password, enterpriseName]
              properties:
                phone: { type: string, pattern: '^1[3-9]\d{9}$' }
                password: { type: string, minLength: 6 }
                enterpriseName: { type: string, maxLength: 100 }
      responses:
        200:
          description: 注册成功
          content:
            application/json:
              schema:
                type: object
                properties:
                  code: { type: integer, example: 200 }
                  data:
                    type: object
                    properties:
                      userId: { type: integer }
                      accessToken: { type: string }
```

**步骤 3：契约锁定**
```powershell
git add docs/api/swagger-v1.0.0.yaml
git commit -m "docs(api): 定版接口契约 v1.0.0"
git push origin develop
```

### 3.3 编码规范检查

**提交前必须执行：**
```powershell

# 4. 单元测试
```

### 3.4 提交代码规范


**提交类型：**
| 类型 | 用途 | 示例 |

| `chore` | 构建/工具 | `chore: 更新依赖版本` |

## 四、接口契约管理

### 4.1 契约变更流程

**场景：需要修改已定义接口**

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ 1.提出变更  │────►│ 2.影响评估  │────►│ 3.审批决策  │
│             │     │             │     │             │
│ 在PR中标记  │     │ 架构师分析  │     │ 影响范围小  │
│ ⚠️接口契约  │     │ 影响范围    │     │ → 快速通过  │
│ 变更        │     │             │     │             │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                              影响范围大 ──────┘
                              → 需协商
                              ▼
                       ┌─────────────┐
                       │ 召集相关    │
                       │ 智能体讨论  │
                       │ 兼容性方案  │
                       └─────────────┘
```

### 4.2 契约变更示例

**步骤 1：更新 Swagger 文档**
```bash
# 修改 docs/api/swagger-v1.0.0.yaml
# 添加新字段或接口
```

**步骤 2：在 PR 中明确标注**
```markdown
## ⚠️ 接口契约变更

### 变更内容
- 在 POST /api/v1/customers 请求中新增 `creditLimit` 字段

### 影响模块
- Agent-CRM（新增字段处理）
- Agent-Frontend（表单添加字段）

### 兼容性
- 向后兼容（新增可选字段）

### 已通知智能体
- [x] Agent-Frontend
```

**步骤 3：通知相关智能体**
```powershell
# 方案 A：使用 GitHub CLI 评论（需先登录 gh）
gh pr comment 123 --body "@agent-frontend @agent-crm 请注意接口变更"

# 方案 B：直接在 GitHub 网页端评论
# 访问 PR 页面，在评论区 @ 相关成员
```

### 4.3 版本升级规则

| 变更类型 | 版本号变化 | 示例 | 是否需要协商 |
|---------|-----------|------|-------------|
| 新增可选字段 | 修订号+1 | v1.0.0 → v1.0.1 | 否 |
| 新增接口 | 次版本+1 | v1.0.0 → v1.1.0 | 否 |
| 新增必填字段 | 次版本+1 | v1.0.0 → v1.1.0 | 是 |
| 删除字段/接口 | 主版本+1 | v1.0.0 → v2.0.0 | 必须 |
| 修改字段类型 | 主版本+1 | v1.0.0 → v2.0.0 | 必须 |

**步骤 1：创建独立的 Migration 分支**


### 5.3 冲突解决

**场景：两个智能体同时修改同一表**

**解决方案：**
1. 架构师协调合并顺序
2. 按优先级执行 Migration
3. 第二个 Migration 基于最新的 Schema 重新生成

```powershell
# 智能体A的 Migration 先合并
# 智能体B需要重新生成 Migration
git checkout develop
git pull origin develop

# 删除旧的 Migration 文件，重新生成
Remove-Item -Path drizzle/0004_agent_b_change.sql -ErrorAction SilentlyContinue
npx drizzle-kit generate:pg --name agent_b_change_v2

# 提交新的 PR
git add .
git commit --amend --no-edit
git push origin migration/agent-b-change --force
```

## 八、GitHub 配置指南

### 8.1 分支保护规则配置

**在 GitHub 仓库 Settings → Branches 中配置：**

**`main` 分支保护规则：**
- ✅ Require a pull request before merging
  - Require 2 approving reviews
  - Dismiss stale PR approvals when new commits are pushed
- ✅ Require status checks to pass before merging
  - Status checks: `Multi-Agent CI`, `migration-check`
- ✅ Require conversation resolution before merging
- ✅ Include administrators

**`develop` 分支保护规则：**
- ✅ Require a pull request before merging
  - Require 1 approving review
- ✅ Require status checks to pass before merging
  - Status checks: `Multi-Agent CI`
- ✅ Require conversation resolution before merging

### 8.2 GitHub Actions 配置

**创建 `.github/workflows/multi-agent-ci.yml`：**

```yaml
name: Multi-Agent CI

on:
  pull_request:
    branches: [develop, main]

jobs:
  detect-agent:
    runs-on: ubuntu-latest
    outputs:
      agent: ${{ steps.detect.outputs.agent }}
    steps:
      - uses: actions/checkout@v4
      - id: detect
        run: |
          BRANCH="${{ github.head_ref }}"
          if [[ $BRANCH == feature/auth* ]]; then 
            echo "agent=auth" >> $GITHUB_OUTPUT
          elif [[ $BRANCH == feature/crm* ]]; then 
            echo "agent=crm" >> $GITHUB_OUTPUT
          elif [[ $BRANCH == feature/inventory* ]]; then 
            echo "agent=inventory" >> $GITHUB_OUTPUT
          elif [[ $BRANCH == feature/finance* ]]; then 
            echo "agent=finance" >> $GITHUB_OUTPUT
          elif [[ $BRANCH == feature/ai* ]]; then 
            echo "agent=ai" >> $GITHUB_OUTPUT
          elif [[ $BRANCH == feature/frontend* ]]; then 
            echo "agent=frontend" >> $GITHUB_OUTPUT
          fi

  backend-quality:
    needs: detect-agent
    if: needs.detect-agent.outputs.agent != 'frontend'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
```

## 九、常见问题解决

### 9.1 合并冲突解决

**场景：两个智能体修改了同一文件**

```powershell
# 1. 拉取最新 develop
git checkout develop
git pull origin develop

# 2. 切换回功能分支
git checkout feature/your-feature

# 3. 合并 develop
git merge develop

# 4. 解决冲突（编辑冲突文件）
# 冲突标记：<<<<<<< HEAD / ======= / >>>>>>> develop

# 5. 标记已解决
git add .
git commit -m "merge: 解决与 develop 的冲突"
```

### 9.2 接口契约不一致

**场景：后端修改了接口，前端未同步**

**解决方案：**
1. 后端智能体在 PR 中明确标注接口变更
2. 使用 GitHub @mention 通知前端智能体
3. 前端智能体更新 Mock 数据和接口调用
4. 双方确认后合并

### 9.3 数据库 Migration 冲突

**场景：两个 Migration 文件序号冲突**

**解决方案：**
1. 使用时间戳命名替代序号（Drizzle 默认）
2. 架构师协调合并顺序
3. 冲突时重新生成 Migration

```powershell
# 查看 Migration 状态
npx drizzle-kit check

# 重新生成（解决冲突后）
npx drizzle-kit generate:pg --name migration_v2
```

### 9.4 CI 失败处理

**步骤 1：查看失败日志**
```powershell
# 在 PR 页面查看 Checks 标签
# 或本地复现
npm run lint
npm run typecheck
npm run test
```

**步骤 2：修复问题**
```powershell
# 修复代码
git add .
git commit --amend --no-edit
git push origin feature/your-feature --force
```

**步骤 3：重新触发 CI**
```powershell
# 推送后会自动触发
# 或在 PR 页面点击 "Re-run jobs"
```

## 十、最佳实践

### 10.1 沟通规范

| 场景 | 沟通方式 | 示例 |
|------|---------|------|
| 接口变更 | PR 评论 + @mention | `@agent-frontend 接口新增字段` |
| 紧急问题 | GitHub Issue | 创建 Bug Issue 并标记 P0 |
| 日常同步 | 每日站会（文字） | 在 Discussion 更新进度 |
| 架构决策 | GitHub Discussion | 创建讨论帖征集意见 |

### 10.2 代码审查清单

**审查他人 PR 时检查：**
- [ ] 代码是否符合编码规范
- [ ] 是否有足够的单元测试
- [ ] 是否影响其他模块
- [ ] 接口契约是否一致
- [ ] 数据库变更是否合理
- [ ] 是否有安全漏洞

### 10.3 性能优化建议

- **数据库查询**：添加必要的索引，避免 N+1 查询
- **API 响应**：使用 Redis 缓存热点数据
- **前端加载**：使用 Next.js 自动代码分割
- **图片资源**：使用 WebP 格式，CDN 加速
